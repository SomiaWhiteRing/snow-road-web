本文件是“面向实现的约束摘要”，不是原始证据本身。
来源分类、证据层与当前实现结构总表见 `Reference/provenance_audit.md`。

按下前进指令后，背景图随机变为剩下两张前进距离会增加1，然后会发生各种事件。

如果前进后的距离是100的倍数，则固定进入BOSS战事件。
除此之外会按 `Random(100)` 的结果进入以下分支：

1. `0..9`：文本展示区从主角的想法里抽取一句展示。10%
2. `10..39`：进入战斗分支。30%
3. `40..49`：捡到火柴。10%
4. `50..54`：发现雪屋。5%
5. `55..94`：什么都没有发生。40%
6. `95`：发现便条。1%
7. `96..97`：发现自动贩卖机。2%
8. `98..99`：发现蜡烛。2%

战斗分支内部还会再判一次：

1. `Random(20) == 0`：进入武器 / 防具遭遇。
2. 否则：从当前 Stage 的怪物池按三角权重抽 1 体。

怪物池不是等概率。若列表长度为 N，则第 1 个怪物权重为 `2N-1`，最后 1 个为 `1`。

额外细节：

1. 火柴事件内部再 `Random(100)`，1% 改为火柴盒，+10 根火柴；否则 +1。
2. 旅馆价格从 1 开始，再独立判定是否乘以 2 / 5 / 10，所以不是均匀随机。
3. 熊警告不是独立事件，而是在第一次真正抽到熊类敌人时，替换掉那次遭遇。
4. 装备遭遇战胜后不是自动换装，而是进入 `〈装备名〉を拾った。 / 装備しますか？` 的确认分支：
   - 选“装備する”才真正应用装备
   - 选“不装备”则直接跳过

故事与关卡切换也有几个必须遵守的原版规则：

1. 开始游戏时不播放 `story00`，标题点击只会进入主界面。
2. BOSS 战后应按“当前 Stage 编号”播放 `storyXX`，不能先手动 `stage += 1`。
3. `stage` 的推进由故事脚本自己控制：
   - 普通 `#end` 默认把 Stage +1
   - 若脚本中已经执行过 `#incStage`，后面的 `#end` 不会再加一次
4. `story09` 的 `#finish` 才是 Another 入口判定点，不是打倒 `coldecot` 后立刻弹框。
5. Another 的确认只会在满足原版条件时出现：
   - 当前仍在 `stage == 9`
   - 当前距离 `<= 1000`
   - 没有 `傷アリ` 标记
6. 若没有进入 Another，不会立刻结束；而是在第 `1001` 步自动播放 `story_close`。
7. 故事 CG 与死亡图的表现层也要按“切页”理解：
   - 原版 `PageMain / PageStory / PageGameOver` 是彼此独立的页面
   - `PageMain` 才是左 300x300 场景区 + 右侧状态区的那一页
   - 故事时会隐藏 `PageMain`、显示 `PageStory`
   - 死亡时会切到 `PageGameOver`
   - 且真正的切页入口是 `0x47972C`，不是战斗 `奥の手` 的 `0x4788FF`
   - `stage >= 10` 的死亡图还会换成 `gameover2.bmp`
   - 因此 Web 端不应把故事 CG / GameOver 继续画成左侧小图，而应统一成覆盖整个游戏画面的 overlay/page
8. 故事脚本里的 `#midi` 不能再忽略：
   - 它是 cutscene 页自己的音乐控制
   - 若参数为空，表示切到无 BGM 状态
   - 因此故事 / CG 时不应继续残留关卡背景音乐
9. 故事脚本的“出页”规则也必须按原版改：
   - 不是“遇到 directive 就 flush 一段文本”
   - 原版是读到**真正的空行**才出一页
   - `#cg / #midi / #wave / #GetName / #incStage` 都只是在当前点击里即时改状态
   - `#notext` 自身不出页；它通常靠后面的空行制造一张“只有 CG、没有文字”的页面
   - 因此像 `story08` 第 69-70 行这种连续正文应并成同一页，而 `story02 / story03` 里被空行分开的句子则要逐句分页
10. `傷アリ` 目前已确认的两条来源：
   - 旅馆里的免费 `奥の手`：先回满 HP / MP，再扣 `ceil(MaxHP / 10)`，并置 `傷アリ`
   - 战斗里的 `奥の手`：仅非 Boss 战可用，按当前 `MaxHP` 直接打伤害，然后扣 `ceil(MaxHP / 10)` 并置 `傷アリ`
11. 战斗 `奥の手` 不是“有限次数技能”，不能再用“剩余 3 次”之类的近似实现。
12. `ドライヴ / フリップ` 也不是“出手即结算一整回合”的技能：
   - 它们是免费动作，不会立刻触发敌方行动
   - `ドライヴ`：临时攻击加 `3..5`；若已学 `オーバードライヴ`，则改为加“当前基础攻击力”
   - `フリップ`：临时防御加 `3..5`；若已学 `フリップフロップ`，则改为加“当前基础防御力”
   - 每次使用后，当前武器 / 防具威力 `-1`
   - 若威力掉到 `0`，对应装备会直接损坏，同时这一路累积的临时加成也会清空
13. `攻撃 / 魔法` 的回合先后手在原版里是 `50% / 50%` 随机，不是固定玩家先手。
14. 只有当整轮结束且战斗仍继续时，原版才会统一衰减：
   - 虚构 MP `-1`
   - `ドライヴ` 累积的临时攻击 `-1`
   - `フリップ` 累积的临时防御 `-1`
15. 战斗不是“点击命令后自动跑完整轮”，而是明确的步进状态机：
   - 点击 `戦う / 魔法` 时，不是只登记命令，也不是立刻结完整轮
   - 命令按钮会先把状态机推进到“当前该展示的第一步”
   - 之后右侧按钮应收束为只剩 `≫`
   - 玩家每点一次 `≫`，再推进一步，直到该轮或该次收束结束
16. 这套步进节奏至少要覆盖下列链路：
   - 普通 `攻撃 / 魔法`：先手方动作 -> 后手方动作 -> 回合末衰减
   - 若某一步已经打死对方：先显示该动作文本，下一步再进入胜败收束
   - `逃げる` 失败：先显示失败文本，再由 `≫` 推进敌方行动
   - 战斗 `奥の手`：`台词 -> 伤害 -> 反动 -> 胜败 / 后续`
   - `≪星舞い≫`：每次 `≫` 只追加一击，不得一次性合并为整串连击
战斗区特效也要按原版分流，不能继续做成“一律整回合结束”或“一律整屏闪一下”：

- `BtnBattleStep (≫)` 的坐标与 `BtnBattle00 (戦う)` 完全重合，都是 `(525, 210)`；不在 `逃げる` 的 `(525, 260)` 上。
- 玩家普通物理命中敌人：
  - 触发的是左侧 `PaintMain` 战斗区里的 `10 tick` 敌图局部闪烁
  - 不是整屏震动，也不是整块白闪
  - 从位图处理方式看，这组闪烁是“红化 / 反相”方向
- 敌人物理命中玩家：
  - 触发的是左侧战斗区 `10 tick` 白闪
- 敌方魔法：
  - `――こごえろ / 氷結`：左侧战斗区淡蓝整块闪
  - `――火葬 / 炎熱`：左侧战斗区纯红整块闪
  - 这组覆盖只发生在 `PaintMain` 的 `300 x 300` 区域内
- 玩家死亡时：
  - 左侧战斗区会先被整块刷白
  - 然后再继续进入原版的 GameOver 切页流程
17. 战斗中的 `火柴 / ライター` 也不应再被当成“玩家出手后敌人立即跟着动”的操作：
   - 它们和 `ドライヴ / フリップ` 一样是免费动作
   - 点击后只发生自身回复 / 消耗 / 文本
   - 不进入敌我回合状态机
18. 敌方“是否施法”的判定也应按原版固定成：
   - 敌人具有施法能力
   - 且当前 MP > 0
   - 且 `Random(2) == 0`
   - 不是按各敌人的自定义概率字段结算
19. 战斗胜利后的结算链也不能再压成一句 summary：
   - 原版至少拆成：
     - `〈敌名〉を倒した！`
     - `〈exp〉点の経験値を得た。`
     - 若可升级：`レベルが上がった！`
     - `最大ＨＰがX点上がった！`
     - 若触发：`最大ＭＰが1点上がった！`
     - 若触发：`攻撃力が1点上がった！`
     - 若触发：`防御力が1点上がった！`
     - 若有装备掉落：`〈装备名〉を拾った。 / 装備しますか？`
     - 若有火柴奖励：`〈n〉本のマッチを手に入れた！`
   - 每条有文本的结算都应是独立一步或独立按钮分支
   - 没有文本的失败分支会在同一次点击里静默递归跳过
20. 升级数值本身也要按原版时序落地：
   - 先判断 EXP 是否足够升级
   - 升级所需 EXP 不是独立表，而是 `当前 LV * 5`
   - 足够则 `EXP -= 所需值`、`LV +1`
   - 然后再依次处理本次升级的 `MaxHP / MaxMP / 攻 / 防`
   - 一轮升级链结束后，若剩余 EXP 仍足够，则继续再跑下一轮升级链
21. 升级不会顺带把当前状态补满：
   - `MaxHP` 增加时，不会把当前 HP 一起加上去
   - `MaxMP` 增加时，也不应把当前 MP 自动回满
22. 胜利结算末尾的“火柴奖励”不是可有可无，而是当前敌人结构 `+0x110` 的精确字段：
   - 装备遭遇：固定 `0`
   - 普通敌人里只有 `おそるべきマッチ = 1`，其余普通敌人都是 `0`
   - BOSS 的原版值分别是：
     - `ゆきのぶ 0`
     - `ゆきの 3`
     - `スノードール 4`
     - `ひらいた花 5`
     - `ベアーハート 7`
     - `ゆきみち 8`
     - `おそるべきエピタフ 10`
     - `だれもいない 0`
     - `クドウ 15`
     - `コルデコット 0`
     - `ルシファー 0`
   - 因此这一步必须真实结算，而不是只留空状态位
23. `types/enemies.ts` 里的 BOSS EXP 也必须切到 exe 数值，不能继续沿用攻略近似：
   - `ゆきのぶ 15`
   - `ゆきの 30`
   - `スノードール 50`
   - `ひらいた花 60`
   - `ベアーハート 80`
   - `ゆきみち 100`
   - `おそるべきエピタフ 120`
   - `だれもいない 0`
   - `クドウ 150`
   - `コルデコット 1000`
   - `ルシファー 10000`

还需要继续坚持下面这些已经坐实的实现约束：

1. `#GetName` 不是注释，而是真的会把玩家名字写进记录开头。
   - `story08` 里会把名字改成 `ララアクス`
2. `原初へ` 不是普通技能 flag：
   - 只在 `stage >= 10` 且当前名字不是 `ララフレア` 时出现
   - 点击后扣 `99 MaxHP`
   - 然后把当前名字直接改成 `ララフレア`
3. 敌方魔法的免疫判定也不是看某个技能是否学过，而是：
   - 当前名字是否已经变成 `ララフレア`
4. 敌方魔法的伤害公式必须改成原版：
   - `damage = max(0, enemyCurrentMp - (playerMp + virtualMp))`
   - 施法名不看敌人配置，而看当前阶段：
     - `stage < 10`：`――こごえろ`
     - `stage >= 10`：`――火葬`
   - 敌人每次施法后自己的当前 MP 还会 `-1`
5. `呪文構築` 不是“每次输入都扣 3 HP”：
   - 进入构筑页时只扣一次 `3 MaxHP`
   - 然后获得 `3` 次 `つぶやく`
   - 每点一次按钮才消耗 1 次
   - 按钮文案应按剩余次数显示：
     - `つぶやく（残り３回）`
     - `つぶやく（残り２回）`
     - `つぶやく（残り１回）`
     - 用尽后 `EXIT`
   - 10 个咒文全部完成后显示 `調べ尽くした`
6. `消し炭になあれ(20)` 的伤害不能只看当前真实 MP：
   - 原版取的是施法前的 `当前真实MP + 虚构MP`
7. `春よりも夏よりもあたたかい日` 的文案和机制也已经不能再用近似实现：
   - 施放时显示 `星がまたたいた……`
   - 状态栏显示 `≪星舞い≫`
   - 它不是一次性写死连击次数，而是开启一个状态
   - 普通攻击时会按“当前点亮的星位数”追加攻击
   - 星位状态应用 `★ / ☆` 表示，而不是继续只存一个“星印数量”
8. 星印与消耗次数必须拆开：
   - 商店买的是星位总数
   - 战斗/地图中真正消耗的是当前点亮数 / `ライター`
