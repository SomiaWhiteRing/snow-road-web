# 雪道原版逆向笔记

本文件整理本轮针对 `original_game/snow/snow.exe` 的静态分析结果，供后续继续复刻时校对。
来源分类与当前实现结构总表见 `Reference/provenance_audit.md`。

## 分析方式

- 使用 `dumpbin` 查看 PE 头与导入表。
- 使用 `Resource Hacker` 抽取 Delphi 窗体资源，确认 UI 结构与事件入口。
- 用 CP932 解码原版 `説明書.html` / `readme.txt`。
- 从 `snow.exe` 中抽取 Shift-JIS 纯文本字符串，补足说明书未写全的功能名与提示语。

## 可确认事实

### 可执行文件

- `snow.exe` 是 32 位 Windows GUI 程序。
- 开发语言与原版 `readme.txt` 一致，为 Delphi。
- 导入了 `winmm.PlaySoundA` 与 `dat/GuruGuruSMF.dll`，说明原版音效与 MIDI 播放都由本体直接驱动。

### Delphi 主窗体结构

从 `FORMMAIN.dfm` 可确认原版至少包含以下页面/面板：

- `PageTitle`
- `PageStory`
- `PageMain`
- `PageSpell`
- `PageGameOver`
- `PanelControl`
- `PanelShop`
- `PanelMagic`

并且主页面上的关键按钮/行为入口与当前 Web 版高度对应：

- `進む`
- `制御`
- `魔法`
- `戦う`
- `逃げる`
- `ドライヴ`
- `フリップ`
- `奥の手`
- `泊まる`
- `セーブ`
- `寄る`

这说明 Web 版继续沿用“主场景 + 覆盖层菜单”的结构是对的。

同时，几个和 CG 展示直接相关的页面尺寸/布局也已经能从 DFM 看清：

- `PageMain`
  - `600 x 400`
  - 左侧 `PaintMain = 300 x 300` 为场景/战斗区
  - 右侧 `ImgMine = 150 x 150` 为角色图
- `PageStory`
  - `600 x 400`
  - `ImgStory` 从 `(0, 0)` 开始承载整张故事 CG
  - `LabStoryTrap` 覆盖底部点击区推进剧情
- `PageGameOver`
  - 独立于 `PageMain / PageStory`
  - 专门承载 `ImgGameOver`

这意味着原版的故事 CG / GameOver 不是“塞进主游戏页左边那块 300 宽场景区”，而是切到独立页面来显示。

## 已核实规则

### 直接攻击公式

原版说明书明确写明，而且本轮也已从 `0x477E08` 的实代码再次坐实：

- 若 `攻击力 > 防御力`，伤害为 `攻击力 - 防御力`
- 若 `攻击力 <= 防御力`，则以 `攻击力 / (攻击力 + 防御力)` 的概率造成 1 点最小伤害，否则 miss
- 若攻击方攻击力为 0，则必 miss

`0x477E08` 的实际伪代码可直接写成：

```text
if (atk - def > 0) {
  return atk - def;
}

if (Random(atk + def) < atk) {
  return 1;
}

return 0;
```

并且调用点也已经能分开：

- 玩家物理攻击 / `≪星舞い≫` 追加攻击
  - `atk = player.currentAttack (+0x44) + temporaryAttackBonus (+0x4C)`
  - `def = enemy.currentDefense (0x489BF8)`
- 敌人物理攻击
  - `atk = enemy.currentAttack (0x489BEC)`
  - `def = player.currentDefense (+0x50) + temporaryDefenseBonus (+0x58)`

所以原版物理伤害确实不是“永远单纯做减法”；只有在 `atk > def` 时才走直减，否则会进入 1 点擦伤概率判定。

### 逃跑

- 说明书明确写明逃跑成功率“约 80%”

### 成长 / Potential

参考资料与攻略页可确认：

- 升级时先掷 `X = 1..6`
- `MaxHP += floor(Potential / 5) + X`
- `Potential += 6 - X`
- 当 `Potential >= 5` 时，有低概率以消耗 5 点 Potential 为代价，提升 `MaxMP / 攻击 / 防御` 之一

### 制御

已从参考资料与 exe 字符串确认的项目：

- `フリップ`，消耗 1
- `ドライヴ`，消耗 1
- `奥の手`，消耗 5
- `フリップフロップ`，前置 `フリップ`，消耗 99
- `オーバードライヴ`，前置 `ドライヴ`，消耗 99
- `攻撃力+1`，消耗 `当前基础攻击力 + 5`
- `防御力+1`，消耗 `当前基础防御力 + 5`
- `MaxMP+1`，消耗 `20`（当 MaxMP 为 0）或 `当前 MaxMP`
- `呪文構築`，消耗 3

其中 `呪文構築` 是本轮通过 exe 字符串和 DFM 控件名重新确认的，之前 Web 版遗漏了这一入口。

本轮继续把制御点击链补齐后，也能确认原版制御面板的成功音效不是 `dat/up.wav`，而是统一的 `dat/hyuun.wav`：

- `BtnCtlLMClick -> 0x47B710 -> 0x472880`
- `BtnCtlATKClick -> 0x47B728 -> 0x4728C4`
- `BtnCtlDEFClick -> 0x47B740 -> 0x472918`
- `BtnCtlMPClick -> 0x47B7F8 -> 0x47296C`
- `BtnCtlWVOKClick -> 0x47B810 -> 0x4729C0`
- `BtnCtlSVOKClick -> 0x47B828 -> 0x472A04`
- `BtnCtlSpellClick -> 0x47B9A4 -> 0x473B1C`
- `BtnCtlFlareClick -> 0x47BB20 -> 0x472A48`
- `BtnCtlWVOK2Click -> 0x47BB38 -> 0x472AA4`
- `BtnCtlSVOK2Click -> 0x47BB50 -> 0x472AE8`

这些底层函数开头都会先取一条 `dat/hyuun.wav` 字符串再调用 `0x5BE3C`，随后才分别执行：

- 扣 `MaxHP`
- 习得技能 flag
- `攻击 / 防御 / MaxMP +1`
- 进入 `呪文構築` 并把剩余次数置为 `3`
- `原初へ` 改名为 `ララフレア`

因此 Web 端若要按原版复刻，制御面板里所有“成功点击并生效”的项目都应播 `hyuun.wav`；`up.wav` 出现在别的系统里，不能混用到制御。

### 商店

本轮继续把 `0x4721D8 -> 0x4722D8 -> 0x4725F4 -> 0x470604` 这条链拆开后，可以确认原版商店并不是“固定四类菜单”，而是**4 个独立商品槽**。每次进入商店时，4 个槽位都会各自重新抽一次：

- 先 `Random(3) + 1` 得到 `variance = 1..3`
- 再 `Random(5)` 决定槽位类型：
  - `0`：武器
  - `1`：防具
  - `2`：本
  - `3`：星印
  - `4`：空槽

其中：

- 武器槽：
  - 名称来自原版武器表 `rank = stage + variance + 1`
  - 价格是 `stage * 2 + variance * 5`
- 防具槽：
  - 名称来自原版防具表 `rank = stage + variance`
  - 价格同样是 `stage * 2 + variance * 5`
- 本：
  - 价格固定 `20`
- 星印：
  - 价格是 `50 + 当前已持有星印数 * 10`
- 空槽：
  - 直接显示 `（品切れ）`

exe 字符串与 `0x4722D8` 的拼接代码进一步确认了商店按钮的真实文案：

- `（品切れ）`
- `（既読）`
- `武器`
- `防具`
- `本`
- `星印`
- `（`
- `）`
- `：`
- `マッチ`

也就是说：

- 武器 / 防具按钮不是只写“武器 / 防具”
- 它会显示成 `〈装备名〉（〈强度〉）：〈价格〉本`
- 本 / 星印则显示成 `〈名称〉：〈价格〉本`
- 若该槽是书且玩家已经读过书，则该按钮直接显示 `（既読）`
- 钱不够时按钮**不会**预先变灰；点下去后才会播放 `dat/bub.wav` 并拒绝购买
- 购买成功则播放 `dat/eat.wav`，把该槽清成 `（品切れ）`

### 地图遭遇 / 前进事件

本轮已直接从 `BtnMainClick -> 0x472F80 -> 0x472ED4/0x472F13` 的代码分支中确认，原版前进事件不是拍脑袋权重，而是 `Random(100)` 的硬编码分段：

- `0..9`：主角独白，10%
- `10..39`：战斗分支，30%
- `40..49`：捡到火柴，10%
- `50..54`：旅馆，5%
- `55..94`：什么都没有发生，40%
- `95`：便条，1%
- `96..97`：自动贩卖机，2%
- `98..99`：存档蜡烛，2%

并且还有几个当前 Web 版之前没还原出来的细节：

- 若前进后的距离是 `100` 的倍数，直接进入 BOSS 战，不走上面的随机分支。
- 火柴事件内部还会再 `Random(100)` 一次：
  - `99%`：+1 根火柴
  - `1%`：火柴盒，+10 根火柴
- 旅馆价格也不是 `1..3` 均匀随机，而是：
  - 先从 `1` 开始
  - `1/2` 概率乘 `2`
  - `1/5` 概率乘 `5`
  - `1/10` 概率乘 `10`
- 熊警告不是独立事件。它只会在“真正抽中熊类敌人时”替换该次遭遇，而且每个 Stage 只触发一次。

### 普通敌人抽选

普通战斗分支内部还不是均匀抽怪，而是用了一个平方随机形成的三角权重。若某 Stage 的候选列表长度为 `N`，则第 1 个敌人的权重最大，为 `2N-1`，最后 1 个敌人的权重最小，为 `1`。

从 `0x4776A0 -> 0x4775AC` 可确认各 Stage 的怪物顺序为：

- Stage 0：`ゆきお`，`アイスフレア`
- Stage 1：`ゆきこ`，`アイスフレア`，`ゆきお`，`白熊`
- Stage 2：`ゆきこ`，`ゆきまさ`，`白熊`
- Stage 3：`ゆきまさ`，`ひらかぬ花`，`エピタフ`
- Stage 4：`白熊`，`白熊`，`白熊王`
- Stage 5：`ゆきひこ`，`ゆきえ`，`ゆきお`，`ゆきこ`，`ゆきまさ`
- Stage 6：`エピタフ`，`おそるべきマッチ`，`白熊王`
- Stage 7：通常敌人为空
- Stage 8：`絶対ゆきお`，`絶対ゆきこ`，`絶対熊`
- Stage Final：`コルデコットオーラ`，`コルデコットオーラ`，`絶対熊`
- Stage Another：`絶対熊`，`フロムエンジン`，`レッドシルエット`

对应成概率后，例子如下：

- Stage 0：`ゆきお 75%`，`アイスフレア 25%`
- Stage 2：`ゆきこ 5/9`，`ゆきまさ 3/9`，`白熊 1/9`
- Stage 5：`ゆきひこ 9/25`，`ゆきえ 7/25`，`ゆきお 5/25`，`ゆきこ 3/25`，`ゆきまさ 1/25`
- Stage 4 / Final 因为列表里写了两次同名敌人，实际也确实提高了该敌人的出现率

### 装备遭遇

战斗分支内部还会先 `Random(20)`：

- 命中 `0` 时，不走普通怪池，而是进入“武器 / 防具遭遇”
- 其余 `19/20` 才走上面的普通敌人抽选

也就是说，装备遭遇的总体概率是：

- `30% * 1/20 = 1.5% / 步`

从 `0x477168` 可确认：

- 会先计算一个 `rank`
- 实际基础值是 `当前 Stage + 1`
- 然后最多进行 4 次独立的 `1/2` 判定，每次成功 `rank + 1`
- `rank` 上限为 `15`
- 再 `1/2` 概率决定是武器还是防具

这里前面一版笔记曾把函数开头的越界检查误读成“基础值 `stage + 5`”。重新核对 `0x477168` 的机器码后，真实流程是：

1. 先计算 `stage + 1 + 4`，只用于和 `15` 比较，做 `UPPERMAX` 越界保护。
2. 真正写入 `rank` 的起点是另一条分支里的 `stage + 1`。
3. 之后才进行 4 次独立 `1/2` 上浮。

所以 `Stage 0` 的装备遭遇 rank 分布其实是：

- `1`：`1/16`
- `2`：`4/16`
- `3`：`6/16`
- `4`：`4/16`
- `5`：`1/16`

这也和实机里出现 `4 / 10 / 4` 的武器怪完全一致，因为那正是 `rank = 4` 时的：

- `HP = 4`
- `ATK = 1 + 2 + 3 + 4 = 10`
- `DEF = 4`

武器 / 防具名都直接来自原版装备表 1~15 档：

- 武器：`手斧` 到 `天地の二等分線`
- 防具：`ケープ` 到 `天幕`

并且装备遭遇的敌方数值也已从代码里确认：

- 经验值：`rank * (rank + 1) / 2`
- HP：`rank`
- 若是武器遭遇：
  - 攻击 = 三角数
  - 防御 = rank
- 若是防具遭遇：
  - 攻击 = rank
  - 防御 = 三角数

这也解释了为什么 Stage 7 虽然“没有敌人”，攻略仍然说武器和防具会出现。

装备遭遇战后的流程也已经进一步核实：

- `BtnEquipNotClick = 0x47c7b8`
- `BtnEquipClick = 0x47c7d8`
- 装备按钮本身不是在按钮事件里直接改数据，而是切换到状态机后续分支
- `0x478c6f` 会显示：
  - `〈装备名〉を拾った。`
  - `装備しますか？`
- 点“装備する”后会进入 `0x478ce2`：
  - 播放 `dat/eat.wav`
  - 显示 `〈装备名〉を装備した！`
  - 调用 `0x470604` 真正应用装备
- 点“不装备”则直接跳过到后续结算状态，不会自动装备

其中 `0x470604` 消费的是保存在 `0x489cdc` 的整份装备记录，而不只是装备名字符串。这说明原版的“装备遭遇”确实是“先掉落，再选择是否换上”，不是战后自动替换。

### 魔法

参考资料已确认以下规则：

- 书买下后，才能进行咒文构筑
- 构筑某个咒文时，需要有不低于其消耗值的真实 MaxMP
- `焦げろ / ゆびさきにともしびを / 手のひらに太陽を` 在 MaxMP 为 0 时也可使用
- 虚构 MP：
  - `ゆびさきにともしびを` 生成 5
  - `手のひらに太陽を` 使其翻倍
  - 移动后归零
  - 战斗中每回合 -1
  - 上限为 `MaxMP * 10`

### 故事脚本 / 关卡衔接

本轮继续从 `LabStartTrapClick -> 0x4737D4`、`LabStoryTrapClick -> 0x475C14`、`0x475A58/0x475B68/0x475BA8` 这条链上确认了原版的故事状态机：

- 标题点击开始时，不会播放任何故事；`0x4737D4` 只是切到 `PageMain` 并隐藏标题页。
- BOSS 战后不是“先 `stage += 1` 再播下一段故事”，而是 `0x478DDC -> 0x475A58(0)`，按**当前 stage 编号**加载 `dat/storyXX.txt`。
  - 这解释了为什么打完第一个 BOSS 播的是 `story00`。
  - 也解释了当前 Web 版原先“开局播 `story00` / BOSS 后播下一关 story”的实现为什么会错位。
- `0x475A58(1)` 会加载 `dat/story_close.txt`，而它不是在拒绝 Another 时立刻触发，而是由前进逻辑在 `距离 == 1001 && stage < 10` 时自动触发。

故事脚本 directive 的真实语义也已基本做实：

- `#end`
  - 走 `0x475BA8`
  - 默认会调用 `0x475B68` 把 `player.stage (+0x120)` 加 1
- `#incStage`
  - 直接走 `0x475B68`
  - 会立刻把 `stage + 1`
  - 同时置位 `0x489BA1`，避免后续 `#end` 再加一次
- `#finish`
  - 先根据当前是否 `stage >= 10` 分流，写出 `dat/after.dat` / `dat/option.dat` 与相关文本
  - 然后只有在以下条件同时满足时，才会弹出进入 Another 的确认框：
    - 当前仍是 `Stage Final`，即 `stage == 9`
    - 当前距离 `<= 1000`
    - `player[+0x6c] == 0`
  - 其中 `player[+0x6c]` 会在 `レポート.txt` 中显示为 `傷アリ`
- `#GetName`
  - 由 `0x475C14` 识别并写入玩家记录
- `#midi` / `#wave` / `#cg` / `#notext`
  - 都是脚本内即时执行的状态变更，不是注释
- 故事分页
  - `0x4710DC` 会先把脚本文本读入内存行表，去掉 `//` 注释，并在末尾补一个 `#End`
  - `0x475C14` 不是“遇到 directive 就出一页”，而是**遇到真正的空行才调用 `0x4759B0` 出页**
  - 因此 `#cg / #midi / #wave / #GetName / #incStage` 都只是即时修改状态，然后继续往下读同一次点击里的后续行
  - `#notext` 本身也只是 no-op；它之所以能制造“空白一页”，是因为后面通常紧跟一个空行
  - 连续正文行会被拼进同一页；被空行隔开的正文则会拆成多次点击
  - 这也解释了为什么 `story08` 里第 69-70 行会同页显示，而 `story02 / story03` 的大部分台词是一句一页
  - 另外，像 `story10` 第 7 行那种“全角空格”并不等同于空行，不会触发分页

故事页的切换方式也已能从代码里进一步坐实：

- `0x4737D4`
  - 标题点击开始后，显式切到 `PageMain`
  - 并隐藏标题页
- `0x475A58`
  - 加载故事脚本后，显式显示故事页并隐藏主游戏页
  - 也就是“切页播放故事”，不是在主页面里局部嵌一个小图
- `0x47C064`
  - 会统一隐藏一批主界面控件
- `0x47972C`
  - 才是真正的 GameOver 切页入口
  - 会先调 `0x45CFF4` 停 MIDI
  - 若 `player[+0x120] >= 10`，则载入 `dat/gameover2.bmp`
  - 否则载入 `dat/gameover.bmp`
  - 然后显示 `PageGameOver`、隐藏 `PageMain`
- `0x4788FF`
  - 不是 GameOver 页入口
  - 它是战斗 `奥の手` 状态 `10` 的台词步骤：播放 `dat/roar.wav`，显示 `もうお仕舞い、ズタズタになあぁぁれええぇぇぇ！！`

结合 DFM，可以把原版表现概括成：

- 主游戏：`PageMain`
- 故事 / CG：`PageStory`
- 死亡：`PageGameOver`

因此 Web 端若要贴近原版，故事 CG 与死亡图应统一成“覆盖整个游戏画面的 cutscene / gameover 页”表现，而不是左侧内嵌 sprite。

`#finish` 后的 Another 流程也已确认：

- 若条件满足且玩家选“是”：
  - 先经 `0x475BA8` 把 `stage` 推进到 `10`
  - 然后显示两条文本：
    - `ふたたび形を成し、目覚めました。`
    - `溶岩から這い出ました。`
- 若条件不满足，或玩家选“否”：
  - 不会立刻关游戏
  - 仍留在 `stage 9 / distance 1000`
  - 下一步走到 `1001` 时才会自动进入 `story_close`
- 因此 `story_close` 是“未进入 Another 的后续收束”，不是 Another 选择框的直接否分支

另外，通关记录函数 `0x4719DC` 里也确认了几个与故事衔接直接相关的标记：

- `player[+0x6c]`：`傷アリ`
- `player[+0x6d]`：`【クリア】`
- `player[+0x6e]`：`【とてつもないものを葬った】`

### `傷アリ` / `奥の手`

本轮把 `傷アリ` 的两条已知来源与战斗 `奥の手` 的完整状态流也继续钉住了：

  - 读取全局中的当前 `MaxHP`
  - 计算 `ceil(MaxHP / 10)`，即 `(MaxHP + 9) / 10`
- 免费旅馆 `奥の手`
  - `BtnINNLMClick = 0x47C788`

  - 最后把 `player[+0x6c] = 1`

  - `BtnBattle02Click = 0x47C758`

### `ドライヴ / フリップ` 与战斗时序

本轮继续拆 `BtnBattle03Click = 0x47C840 -> 0x4792C0` 与 `BtnBattle04Click = 0x47C858 -> 0x479430`，并把它们与主战斗状态机 `0x4784BC` 拼起来后，可以确认：

- `ドライヴ` 与 `フリップ` 都不是“立刻结算敌我一回合”的按钮
- 它们是免费动作：
  - 点击后直接改玩家临时战斗参数
  - 不会立刻触发敌方行动
  - 也不会立刻进入胜负判定
- `ドライヴ`
  - 播放 `dat/hyuun.wav`
  - 若 `player[+0x60] == 1`（已习得 `オーバードライヴ`），则：
    - `tempAtk += baseAtk`，其中 `baseAtk = player[+0x48]`
  - 否则：
    - `tempAtk += Random(3) + 3`，也就是 `+3..5`
  - 然后 `weaponPower = player[+0xBC] - 1`
  - 显示 `〈武器名〉を駆動した。`
  - 若武器威力掉到 `0` 以下：
    - 追加显示 `〈武器名〉は砕け散った。`
    - 调 `0x4704F8(player + 0x70)` 清空武器结构
    - 该调用会同时把 `tempAtk (+0x4C)` 一并清零
- `フリップ`
  - 结构完全对称于 `ドライヴ`
  - 若 `player[+0x61] == 1`（已习得 `フリップフロップ`），则：
    - `tempDef += baseDef`，其中 `baseDef = player[+0x54]`
  - 否则：
    - `tempDef += Random(3) + 3`
  - 然后 `armorPower = player[+0x10C] - 1`
  - 显示 `〈防具名〉を翻した。`
  - 若防具威力掉到 `0` 以下：
    - 追加显示 `〈防具名〉は細切れになった。`
    - 调 `0x4704F8(player + 0xC0)` 清空防具结构
    - 该调用会同时把 `tempDef (+0x58)` 一并清零

而 `攻撃 / 魔法` 所走的整回合时序也已能明确写出：

- `BtnBattle00Click` 会把双方行动标记都置 1，并把玩家命令字符串写成 `攻撃`
- 主状态 `case 1` 在“玩家待行动 / 敌人待行动”同时为真时，会 `Random(2)`：
  - `50%` 先执行玩家
  - `50%` 先执行敌人
- 因此原版的普通攻击与战斗中施放魔法，都不是固定“玩家先手”
- 只有当双方这一轮都完成、且战斗仍未结束时，才会调用 `0x477434`
- `0x477434` 会统一衰减这些临时值各 `1` 点：
  - `virtualMp = player[+0x40]`
  - `tempAtk = player[+0x4C]`
  - `tempDef = player[+0x58]`

因此也可以确认：

- `ドライヴ / フリップ` 可以在不开启敌方行动的前提下预先堆叠
- 它们的临时加成不会在“被打一次 / 打一次”后立刻清空，而是只会在整轮结束时衰减 `1`
- `オーバードライヴ / フリップフロップ` 的真实效果不是“倍率翻倍”，而是把原本的固定 `3..5` 增量改成“当前基础攻 / 防”

### 战斗步进按钮与状态推进

本轮又把几个按钮入口与 `0x4784BC` 的分派关系继续钉实，可以更准确地回答“原版是不是点完命令后进入 `≫` 逐步结算”这个问题：

- `BtnBattle00Click = 0x47C65C`
  - `xor eax, eax`
  - `call 0x479620`
  - `call 0x479168`
  - `call 0x47B9F8`
- `BtnBattleStepClick = 0x47C68C`
  - `xor eax, eax`
  - `call 0x479620`
  - `call 0x4784BC`
  - `call 0x47B9F8`

其中：

- `0x479168`
  - `489D38 = 1`
  - `489D39 = 1`
  - 命令字符串写成 `攻撃`
  - 然后走 `0x479130(1)`
- `0x4791A0`
  - 结构对称于 `0x479168`
  - 命令字符串写成 `魔法`
  - 记录咒文 index
  - 然后也走 `0x479130(1)`
- `0x479130`
  - 会把 `489D34 = state`
  - 然后**立刻调用** `0x4784BC`

这说明原版不是“命令按钮只登记输入，不做任何结算”，也不是当前 Web 版旧实现那样“一次点完直接跑完整轮”。
真实行为是：

1. 玩家点 `戦う / 魔法 / 奥の手 / 逃げる` 等命令。
2. 按钮事件会先把战斗状态推进到**当前应展示的第一步**。
3. 随后界面收束为 `≫`。
4. 玩家每点一次 `≫`，`BtnBattleStepClick` 再把状态机推进一步，直到该轮或该次结算链真正结束。

`0x47C064` 也再次支持了这一点：

- 它会先把一串战斗侧按钮统一 `Visible = false`
- 多个战斗 case 都是先 `call 0x47C064`
- 然后只把 `form + 0x30c` 再单独显示
- `FORMMAIN.dfm` 中 `BtnBattleStep` 正是那个 caption 为 `≫` 的按钮

因此用户对“原版会进入只剩 `≫` 的逐步结算状态”的判断是**大方向正确**的，但要补一句更精确的话：

- 原版是“命令点击后先立刻跑出第一步，然后交给 `≫` 继续逐步推进”
- 不是“点完命令后完全不结算，只排队等下一步”

对 `0x4784BC` 来说，本轮已能稳定写出的关键 case 顺序是：

- `case 1`
  - 若玩家 HP `<= 0`：转入失败收束分支，并继续显示 `≫`
  - 若敌 HP `<= 0`：转入胜利收束分支，并继续显示 `≫`
  - 若双方都待行动：`Random(2)` 决定玩家先手还是敌先手，然后**同一次调用里**立刻进入 `case 2` 或 `case 3`
  - 若只剩一方待行动：同样会在同一次调用里直接进入对应 action case
  - 若双方都已行动完：调用 `0x477434` 做整轮末衰减
- `case 2`
  - 玩家动作执行
  - `攻撃` / `魔法` 都在这里真正生效
  - 执行完后不会继续把整轮跑完，而是把状态改回 `1` 等待下一次 `≫`
- `case 3`
  - 敌方动作执行
  - 若敌可施法、当前 MP > 0 且 `Random(2) == 0` 则施法，否则物理攻击
  - 执行完后同样回到 `1`
- `case 5`
  - `≪星舞い≫` 追加攻击链
  - 每次 `≫` 推进一击
  - 直到附加攻击耗尽后，才回到 `case 1`
- `case 10 -> 11 -> 12`
  - 战斗 `奥の手` 的三步链
- `case 20`
  - 逃跑失败文本
  - 失败后不会整轮自动结算，而是继续交给 `≫`

另外两条容易被当前实现带偏的点，本轮也已经可以归入“逆向确认”：

- `BtnChasClick -> 0x473450`
  - 直接处理回血 / 扣火柴 / 文本
  - **不进入**战斗状态机
- `BtnLighterClick -> 0x473620`
  - 直接处理燃料 / 回复 / 文本
  - **不进入**战斗状态机

因此战斗中 `火柴 / ライター` 与 `ドライヴ / フリップ` 一样，都应按“免费动作，不触发敌方行动”理解，而不是当前旧 Web 版那样把它们塞进整轮结算。

### 战斗画面绘制 / 命中特效

本轮把 `0x47A9F0 -> 0x47A6D0` 这条绘制链继续拆开后，已经可以把“原版战斗里到底闪什么”写得更具体：

- `0x47A9F0`
  - 会预先创建 7 张 `300 x 300` 的 `TBitmap`
  - 全局分别落在：
    - `0x489D54`：主离屏缓冲
    - `0x489D58`：当前战斗底图
    - `0x489D5C / 0x489D60 / 0x489D64`：当前遭遇图的处理层
    - `0x489D68 / 0x489D6C`：整块颜色覆盖层
- `0x47A9F0` 还会把两张整块颜色图直接填成固定 24-bit BGR：
  - `0x489D68 = FF 80 80`，也就是偏淡的蓝色，约等于 `#8080FF`
  - `0x489D6C = 00 00 FF`，也就是纯红，`#FF0000`
- `0x47A6D0`
  - 是战斗区的统一刷新函数
  - 每次先把 `0x489D58` 复制到 `0x489D54`
  - 然后按几个 battle effect flag 继续往 `0x489D54` 叠图
  - 最后再一次性 blit 到 `PaintMain`

因此原版这里的“攻击特效”本质是**左侧 `PaintMain` 这块 300x300 战斗区里的位图重绘**，不是单独移动按钮，也不是通过坐标抖动做 screen shake。

已经能坐实的几组效果如下：

- `0x479D54`
  - 会把当前遭遇图载入 `0x489D5C`
  - 再预处理出 `0x489D60 / 0x489D64`
  - 最后把 `0x47E98C = 1`
- `0x478F84`
  - 会把 `0x47E98C = 0`
- `0x47E98C`
  - 因而就是“当前战斗区里是否存在这组可闪烁的遭遇图层”

- `0x479D44`
  - 置 `0x47E990 = 10`
  - 同时把参数写到 `0x47E994`
- `0x47A6D0` 读取这组值时：
  - 不会整块白闪
  - 而是把 `0x489D5C / 0x489D60 / 0x489D64` 这组三层图按 ROP 重新叠到战斗图上
  - 从 `0x479D54` 对像素的预处理方式看，这里走的是**红化 / 反相方向的局部闪烁**
  - 能坐实的是“普通物理命中时，闪的是当前遭遇图层，不是整个屏幕或整个窗口”；至于肉眼上更像“发红”还是“负片闪”，取决于这组 ROP 叠加后的最终观感
- `0x477E40`
  - 玩家普通物理命中敌人时会：
    - 播 `dat/dun.wav`
    - `xor eax, eax ; call 0x479D44`
  - 所以玩家普通攻击命中后的特效就是这组 **10 tick 的敌图局部闪烁**
- `0x478210` 那条“对敌造成伤害并播 `dat/fire.wav`”的分支
  - 会 `mov al, 1 ; call 0x479D44`
  - 这说明它和普通物理命中共用同一套 10 tick 机制
  - 但 `al = 1` 会让奇数 tick 再额外叠一次 `0x489D64`
  - 因而是更强的一档局部闪烁

- `0x479CF0`
  - 置 `0x47E998 = 10`
  - 同时把参数 shortstring 写到 `0x47E99C`
- `0x47A6D0` 对这组值的处理是：
  - 只在 odd tick 生效
  - 若 `0x47E99C == 氷結`
    - 直接把整张 `0x489D68` 盖到战斗区
    - 也就是 **淡蓝整块闪**
  - 若 `0x47E99C == 炎熱`
    - 直接把整张 `0x489D6C` 盖到战斗区
    - 也就是 **纯红整块闪**
  - 其余字符串
    - 走 `0x00FF0062 = WHITENESS`
    - 也就是 **整块白闪**
- `0x477FA0`
  - 敌人物理命中玩家时会：
    - 播 `dat/zurl.wav`
    - `xor eax, eax ; call 0x479CF0`
  - 因此敌人物理命中玩家触发的是 **10 tick 的战斗区白闪**
- `0x478280`
  - 敌方施法时会把 `氷結 / 炎熱` 传给 `0x479CF0`
  - 因此敌方魔法的画面效果已经可以直接定性为：
    - `――こごえろ`：淡蓝整块闪
    - `――火葬`：纯红整块闪

还有两组辅助绘制 flag 也已经能定位：

- `0x479CE0`
  - 直接写 `0x47E9A0`
- 进入战斗时
  - `0x47652B -> mov al,1 ; call 0x479CE0`

    - `mov al,1 ; call 0x479CE8`

  - 所以死亡前的战斗区会先被**整块刷白**

### 胜利结算 / 经验 / 升级链

本轮继续把 `state 0x65` 之后的胜利链拆开了。`0x4784BC` 在处理 `state > 0x65` 时，不是落回普通回合，而是进入另一张以 `0x66` 为基底的 jump table：

- 入口逻辑在 `0x478584` 一带
- 会先做：
  - `eax -= 0x66`
  - `cmp eax, 0x1c`
  - `jmp dword ptr [eax*4 + 0x478597]`
- 目前已确认的关键状态为：
  - `0x65 -> 0x478A17`
  - `0x66 -> 0x478A53`
  - `0x67 -> 0x478AB2`
  - `0x68 -> 0x478B41`
  - `0x69 -> 0x478BA8`
  - `0x6A -> 0x478C04`
  - `0x6B -> 0x478C60`
  - `0x6E -> 0x478C6F`
  - `0x6F -> 0x478CE2`
  - `0x78 -> 0x478D59`
  - `0x82 -> 0x478DCC`

对应的真实结算顺序已经能写成：

1. `case 1`
   - 发现敌 HP `<= 0`
   - 播 `dat/break.wav`
   - 显示 `〈敌名〉を倒した！`
   - 把状态设成 `0x65`
   - 切到 `≫`
2. `state 0x65 @ 0x478A17`
   - 把 `enemyExp (0x489BD4)` 加到玩家 `EXP (+0x2C)`
   - 显示 `〈数值〉点の経験値を得た。`
3. `state 0x66 @ 0x478A53`
   - 调 `0x47611C` 取“下一次升级所需 EXP”
   - `0x47611C` 本体只有一条核心计算：
     - `return player.LV * 5`
   - 若当前 EXP 不足：
     - 直接 `0x479130(0x6E)`，也就是**同一次点击里**跳去后续掉落/收尾链
   - 若当前 EXP 足够：
     - `EXP -= 所需值`
     - `LV (+0x24) += 1`
     - 播 `dat/open.wav`
     - 显示 `レベルが上がった！`
4. `state 0x67 @ 0x478AB2`
   - 先掷 `X = Random(6) + 1`
   - `MaxHP (+0x30) += floor(Potential / 5) + X`
   - `Potential (+0x28) += 6 - X`
   - 显示 `最大ＨＰがX点上がった！`
   - 关键点：这里只改 `MaxHP`，**没有**给当前 HP `(+0x34)` 同步回满或补涨
5. `state 0x68 @ 0x478B41`
   - 若 `MaxMP (+0x38) > 0`、`Potential >= 5` 且 `Random(20) == 0`
     - `Potential -= 5`
     - `MaxMP += 1`
     - 显示 `最大ＭＰが1点上がった！`
   - 否则直接 `call 0x4784BC`
     - 即**同一次点击里**静默跳去下一个升级判定，不额外停顿
6. `state 0x69 @ 0x478BA8`
   - 若 `Potential >= 5` 且 `Random(20) == 0`
     - `Potential -= 5`
     - `攻击力 (+0x48) += 1`
     - 显示 `攻撃力が1点上がった！`
   - 否则同样静默递归到下一状态
7. `state 0x6A @ 0x478C04`
   - 若 `Potential >= 5` 且 `Random(20) == 0`

   - 也就是在一轮升级链结束后，**同一次点击里**重新检查剩余 EXP 是否还能再升一级

这条链能直接推翻当前 Web 版之前的几个合并实现：

- 经验文本不是和“击倒文本”同一步
- 升级不是一次性黑箱结算，而是 `レベルが上がった！ -> 最大HP -> 可能的 MaxMP/攻/防` 的独立步进链
- 升级时若某条 `5%` 增长没触发，原版会在**同一次点击里静默跳过**，不会额外浪费一步
- 升级时不会自动回复当前 HP / MP，只会提升对应的上限或基础值
- 掉装备确认与战斗结束清理之间，还存在一个“获得火柴”的独立状态 `0x78`

### `#GetName` / 真名字段

### `原初へ`

控制面板刷新函数 `0x473CD8` 已确认 `原初へ` 的显示条件不是“Another 资格”而是：

- 当前 `stage >= 10`
- 且玩家记录开头 shortstring **不等于** `ララフレア`

点击 `原初へ` 的事件入口是：

- `BtnCtlFlareClick = 0x47CB20 -> 0x473A48`

其真实效果已经坐实：

- 扣 `99` 点 `MaxHP`
- 把玩家名字 shortstring 直接改成 `ララフレア`

也就是说，原版里“魔法完全耐性”的本质不是普通技能 flag，而是：

- 玩家当前名字 `== ララフレア`

### 敌方魔法

敌方施法函数是 `0x478280`，本轮已确认：

- 若当前 `stage < 10`，施法名是 `――こごえろ`
- 若当前 `stage >= 10`，施法名是 `――火葬`
- 伤害不是看敌人魔法类型表，而是：
  - 先取 `enemyCurrentMp`
  - 再减去 `player.currentMP + player.virtualMP`
  - 最终伤害为 `max(0, enemyCurrentMp - (playerMP + virtualMP))`
- 敌人每次施法后，自己的当前 MP 还会 `-1`

若玩家名字 shortstring 已是 `ララフレア`，则直接走“完全无效”分支：

- 显示 `の魔法は効かなかった！`
- 不再结算上述伤害

### `つぶやく` / 咒文构筑

`BtnBuildSpellClick = 0x47C9D0` 这一链已经拆清：

- `呪文構築` 本身只在进入构筑页时扣一次 `3 MaxHP`
- 同时把剩余次数 `489B94` 置为 `3`
- 之后每点一次 `BtnBuildSpell`
  - 先把剩余次数 `-1`
  - 再拿输入框文本与 10 个完整咒文名逐一比较
  - 若命中一个“尚未习得且 MaxMP 条件满足”的咒文，就播放对应 `dat/spellNN.wav` 并习得
  - 否则播放 `dat/bub.wav`

构筑页按钮文案也已确认：

- `つぶやく（残り３回）`
- `つぶやく（残り２回）`
- `つぶやく（残り１回）`
- 用尽后变成 `EXIT`
- 全 10 个咒文都构筑完后改成 `調べ尽くした`

这说明原版并不是“每次输入都重新付 3 HP”，而是“一次进入，最多 3 次 whisper”。

### `≪星舞い≫`

`春よりも夏よりもあたたかい日` 的效果本轮也已经钉住：

- `0x475730 -> 0x479158`
- 会把全局 byte `0x47E978` 置为 `1`
- 同时显示 `星がまたたいた……`

状态面板里出现 `≪星舞い≫` 的唯一判定就是读取这个 flag：

- `0x479160` 返回 `byte ptr [0x47E978]`

战斗中的真实行为也基本坐实：

- 普通攻击后若该 flag 已置位：
  - 进入后续状态 `5`
  - 会按 `player[+0x114]` 的当前值继续追加普通攻击
- 因此它不是“一次性写死连击数”的效果
- 更接近“星舞い状态开启后，普通攻击会追加 `当前点亮星位数` 次攻击”

结合报告函数可推得：

- `player[+0x118]` 是星位总数
- `player[+0x114]` 是当前点亮中的星位 / 可消耗燃料
- 状态展示会用 `★ / ☆` 输出当前点亮情况
