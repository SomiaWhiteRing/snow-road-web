## 两个 `消し炭になあれ`

这是原版里两个独立的 spell index，不是同一个咒文的两种显示。

| index | 完整名 | 短标签 | cost | 实际 power | 区别 |
| --- | --- | --- | ---: | --- | --- |
| 1 | `消し炭になあれ` | `消炭` | 1 | 5 | 低成本固定威力版 |
| 3 | `消し炭になあれ` | `消炭` | 20 | `施放前当前真实 MP + 虚构 MP` | 高成本、随当前 MP 总量变化的版 |

关键点：

- 原版完整名数组和短标签数组里，它们都完全同名。
- 原版 UI 不是靠名字区分它们，而是靠内部 index、cost 和结算函数区分。
- 因为构筑函数按 `index 0 -> 9` 顺序扫描，所以同样输入 `消し炭になあれ` 时：
  - 如果 index 1 还没学，且当前可用 MP 满足条件，会先学到 index 1。
  - 只有 index 1 已学过之后，完全相同的输入才会继续落到 index 3。

## 学习魔法的真实条件

原版的学习条件不是“MaxMP 足够”，而是“当前这一刻能不能按该咒文的规则施放它”。

`0x474218` 的逻辑可直接写成：

```text
if (spell.requiresRealMpOnly) {
  return currentMp >= cost
}

return currentMp + virtualMp >= cost
```

也就是说：

- `己ならざるのぬくもりよ / 春ノ夢 / 夏ノ空`
  - 学习时只看当前真实 MP。
- 其余咒文
  - 学习时允许用“当前真实 MP + 虚构 MP”通过判定。
- 所以旧结论“构筑看 MaxMP”是错的。

## 原版咒文构筑流程

`0x473F58` 的流程已经能稳定还原为：

1. 必须先买过书。
2. 进入 `呪文構築` 时，只扣一次 `3 MaxHP`。
3. 同时把剩余 `つぶやく` 次数置为 `3`。
4. 每点一次 `つぶやく`：
   - 先把剩余次数减 `1`
   - 读取输入框文本
   - 与 10 个完整咒文名按 index 顺序逐一比较
   - 找到第一个“文本相等、尚未习得、当前可用 MP 条件满足”的 spell 就成功
   - 成功时播放对应的 `spellNN.wav` 并置 learned flag
   - 全部失败则播放 `bub.wav`

补充：

- 原版是 exact match，不是模糊匹配。
- `spellNN.wav` 这组音效用于“构筑成功”，不是正常施法音效。
- 全 10 个咒文都学会后，按钮文案会变成 `調べ尽くした`。

## `≪星舞い≫` 的真实行为

`春よりも夏よりもあたたかい日` 对应 `index 9`，其效果不是直接打出一串连击，而是先置一个状态 flag。

已经确认的链路：

- `0x475730`
  - 置全局 byte `0x47E978 = 1`
  - 显示 `星がまたたいた……`
  - 播放 `dh.wav`
- `0x479160`
  - 读取这个 flag，决定状态面板是否显示 `≪星舞い≫`
- 战斗状态机 `case 5`
  - 普通攻击后，若该 flag 已置位，则进入追加攻击链
  - 每点一次 `≫` 追加一击
  - 追加次数取自 `player[+0x114]`，即当前点亮中的星位数

因此总攻击次数应理解为：

- 普通攻击本身 `1` 次
- 再追加 `当前点亮星位数` 次
- 合计 `1 + 当前点亮星位数`

这也和原版资料页里“`１＋星印の数だけ連続攻撃できるようになる`”的描述一致。结合报告函数可进一步对应为：

- `player[+0x118]`：星位总数
- `player[+0x114]`：当前点亮中的星位 / 可消耗燃料

## 对 Web 复刻的直接约束

按以上逆向结果，Web 端必须满足：

- 咒文构筑按 `SPELLS` 原顺序扫描，不能把两个 `消し炭になあれ` 合并。
- 学习判定必须走“当前可用 MP”而不是 `maxMp`。
- `春よりも夏よりもあたたかい日` 不能在场外施放。
- `己ならざるのぬくもりよ / 春ノ夢 / 夏ノ空` 的学习与施放都只能用当前真实 MP。
